# v22 원인 분석 및 수정 내역

## 확인한 첨부 파일 구조

- 근무계획 원본과 실제 근태 RAW의 사번은 대부분 정상 매칭되었습니다.
- 첨부된 6월 최종본의 `출근 미등록` 시트 머리글은 `근태수정발생일`이며 증빙 입력 열은 K열 `증빙여부(O 입력)`입니다.
- 현재 전달된 최종본 파일에는 데이터 행의 K열 O가 없어 실제 O 입력 사례 자체는 재현할 수 없었습니다. O가 입력된 수정 최종본을 별도로 선택하면 아래 수정 로직이 적용됩니다.

## v21에서 O가 불안정했던 원인

1. 날짜 머리글 목록에 실제 최종본의 `근태수정발생일`이 없어서 고정 열 위치 보조 판정에 의존했습니다. 열 추가·이동 또는 행 수가 적은 수정본에서는 날짜를 찾지 못할 수 있었습니다.
2. 증빙 O가 확인되면 동일 사번·일자의 `missingRows`, `unexpectedRows`, `mismatchRows`를 모두 제거했습니다. 이 때문에 계획이 휴무·연차인데 실제 출근한 건까지 계획 상이에서 사라졌습니다.
3. 월 마감에 O 날짜와 일별 계획·근태를 별도로 저장하지 않아, 나중에 부여 설정을 추가하거나 재계산할 때 발생일의 계획 우선 판정을 완전히 복원할 수 없었습니다.

## v21에서 발생일 계산이 반영되지 않던 원인

- 웹 분석 시점에는 계획 우선 판정을 했지만 D1에는 최종 대상 날짜만 저장했습니다.
- 발생일 부여를 월 마감 후 추가·수정하면 서버는 해당 날짜가 계획 근무인지, 계획 휴무·미출근인지, 계획 휴무·실제 출근인지 다시 판단할 원자료가 없었습니다.
- 미래 발생분은 사용 시작일부터 선반영되었지만, 발생일이 지난 뒤 미부여 대상으로 확정될 때 과거 사용월까지 lot를 소급 제외하지 않고 남은 수량만 종료했습니다.

## v22 수정 결과

- `근태수정발생일` 직접 인식 및 머리글 기반 열 탐색 강화
- 단순 출근 미입력만 제거하고 휴무·휴가 출근 상이는 유지
- 증빙 O 날짜의 `상담사근태` 값을 `출근`, 채우기 `#A9D08E`로 최종 적용
- 월 마감에 `daily_statuses_json`, `evidence_dates_json`, `base_allowance_raw`, 발생일 휴무 추가 내역 저장
- 발생일 확정 후 계획·근태 기준으로 부여 lot를 사용 시작월까지 소급 재계산
- 같은 경로·같은 월 교체 저장 및 기존 D1 원본 보존 방식 유지
- `계획&근태 상이 인원`을 네이비 대시보드형으로 전면 개편

- 발생일 판정 단위검증 통과

  - 7월 발생일 휴무·미출근 확정: 6월 사용분까지 소급 제외하고 7월 기본 휴무 +1
